# ODESM

`ODESM` (Orbital Differential Equation Solver Module) - ранняя версия Python-движка для моделирования орбитальных миссий спутников.

Проект находится в активной разработке и пока не является production-ready. API и структура файлов могут меняться.

## Статус проекта

- Текущий фокус: численная пропагация траектории в `ECI` и базовые модели тяги/расхода топлива.
- Есть рабочий класс `Propagator` в `DESM_engine.py`.
- Есть каркас для миссионного уровня (`MissionSimulator`) в `simulation.py`.
- Планируемое направление: полноценный движок сценариев орбитальных миссий (множество маневров, этапы миссии, события, постобработка).

## Система координат и физические допущения

- Все вычисления выполняются в `ECI` (Earth-Centered Inertial).
- `ECI` фиксирована относительно звезд и не учитывает вращение Земли напрямую.
- Варианты с `J2` и `J3` учитывают несферичность геопотенциала и косвенно влияют на форму орбиты.
- Для задач, где нужна связь с Землей (например, широта/долгота, наземный трек), рекомендуется использовать `astropy` для преобразований `ECEF <-> ECI`.

## Структура проекта

- `DESM_engine.py` - численный движок и класс `Propagator`.
- `objects.py` - модели сущностей (`Engine`, `Satellite`, `Maneuver`, `Trajectory`).
- `simulation.py` - каркас миссионного уровня (`MissionSimulator`).
- `config.py` - физические константы (`mu`, `R_e`, `J2`, `J3`, `g0`).

## Основной API на текущем этапе

### Класс `Propagator`

Импорт:

```python
from ODESM_engine import Propagator
```

Создание:

```python
propagator = Propagator(amendments='J2')
```

Параметр `amendments`:
- `'None'` - центральное поле Земли;
- `'J2'` - центральное поле + гармоника `J2`;
- `'J3'` - центральное поле + гармоники `J2` и `J3`.

### Метод `predict_trajectory(state_vector, duration, dry_mass)`

Входные аргументы:
- `state_vector` - список/кортеж/`np.ndarray` из 11 элементов:

```text
[x, y, z, vx, vy, vz, Fx, Fy, Fz, m, Isp]
```

где:
- `x, y, z` - м,
- `vx, vy, vz` - м/с,
- `Fx, Fy, Fz` - Н,
- `m` - кг,
- `Isp` - с.

- `duration` - длительность моделирования в секундах (`duration > 0`).
- `dry_mass` - сухая масса спутника в кг (`dry_mass > 0` и `m0 > dry_mass`).

Возвращаемое значение:

```text
[t, x, y, z, vx, vy, vz, Fx, Fy, Fz, m, Isp]
```

все элементы - массивы `numpy` одинаковой длины.

## Логика расхода топлива

В текущей реализации:
- сначала интегрирование идет с заданной тягой;
- если масса достигает `dry_mass` раньше конца интервала, интегрирование продолжается до конца времени с нулевой тягой;
- после выработки топлива масса фиксируется на `dry_mass`.

Это позволяет рассчитывать траекторию на весь заданный интервал даже при раннем окончании топлива.

## Минимальный пример

```python
import numpy as np
from config import R_e, mu
from ODESM_engine import Propagator

state_vector = [
    R_e + 400e3,  # x
    0.0,  # y
    0.0,  # z
    0.0,  # vx
    float(np.sqrt(mu / (R_e + 400e3))),  # vy
    0.0,  # vz
    80.0, 0.0, 0.0,  # Fx, Fy, Fz
    500.0,  # m
    300.0,  # Isp
]

propagator = Propagator(amendments='J2')
t, x, y, z, vx, vy, vz, Fx, Fy, Fz, m, Isp = propagator.predict_trajectory(
    state_vector=state_vector,
    duration=1800.0,
    dry_mass=450.0,
)

print(f"Samples: {len(t)}")
print(f"Final mass: {m[-1]:.3f} kg")
```

## Миссионный уровень (в разработке)

`MissionSimulator` в `simulation.py` пока является каркасом. В следующих итерациях планируется:
- сценарий с несколькими маневрами подряд;
- управление профилем тяги во времени;
- накопление траектории в `Trajectory`;
- единый интерфейс для запуска полной миссии.

## Ограничения текущей версии

- Нет завершенной системы событий миссии (вход/выход из тени, окна связи, ограничения по орбите и т.д.).
- Нет полноценного high-level API для последовательности маневров.
- Нет полной валидации совместимости параметров спутника/двигателя на уровне миссии.
- Возможны изменения сигнатур и структуры данных в следующих версиях.

## Что можно использовать уже сейчас

- `Propagator` как low-level интегратор динамики в `ECI`.
- `objects.py` для описания спутника, двигателя и маневров.
- `config.py` как единый источник физических констант для расчетов.
